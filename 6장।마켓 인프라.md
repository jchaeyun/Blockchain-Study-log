# 6장।마켓 인프라



## 1. 마켓 인프라의 현황과 비트코인의 지표성

암호화폐 시장은 초기 인프라 부족으로 인해 매우 불안정했으나 현재는 구조적 지원이 늘어난 상태다. 하지만 여전히 규제 미비와 시세 조작 문제가 존재하며, 거래의 60~80%가 투기성 거래로 추정된다.

* **BTC의 영향력:** 비트코인은 전체 암호화폐 시장의 가격 지표다. 대부분의 암호화폐는 비트코인의 가격 추세를 추종한다.
* **거래 방식의 진화:** 초기 P2P 대면 거래나 채굴 보상을 통한 취득에서 현재는 암호화폐 ATM과 전문 거래소를 통한 거래가 주류를 이룬다.



## 2. 암호화폐 거래소의 유형과 매매 구조

개발자는 거래소의 백엔드 로직과 오더북 관리 방식을 이해해야 한다.

* **거래소 분류:**
    * **중앙집중화 거래소(CEX):** 기업이 운영하며 수수료를 징수한다. 속도가 빠르나 자산 보관(Custody) 리스크가 존재한다.
    * **탈중앙화 거래소(DEX):** 스마트 컨트랙트로 오더북을 운영한다. 자산은 사용자가 직접 보관하며 투명성이 높다.
    * **현물 vs 파생상품:** 실제 자산을 소유하는 현물 거래소와 레버리지, 옵션, 선물 등 복잡한 금융 도구를 제공하는 파생상품 거래소로 나뉜다.
* **핵심 거래 용어:**
    * **오더북(Order Book):** 미결제 매수/매도 주문의 시각적 리스트다. 암호화폐는 소수점 이하 8자리까지 분할 가능하므로 데이터 처리 시 정밀도가 중요하다.
    * **슬리피지(Slippage):** 주문 시 예상 가격과 실제 체결 가격의 차이다. 유동성이 부족한 시장에서 대량 주문 시 발생하며, 이는 오더북의 깊이(Depth)와 직결된다.
    * **메이커/테이커(Maker/Taker):** 유동성을 제공하는 주문(메이커)과 기존 주문을 체결하는 주문(테이커)에 대해 수수료를 차등 부과하는 모델이다.





## 3. 마켓의 비정상적 움직임과 조작

* **자금세탁(Wash Trading):** 인위적인 거래량을 발생시켜 시장을 왜곡하는 행위다. 거래량 부풀리기를 통해 거래소 순위를 높이거나 가격을 조작하는 데 악용된다.
* **고래(Whale):** 막대한 자산을 보유한 헤비 트레이더다. 이들의 자금 이동은 시장에 큰 파장을 일으키며, 특정 가격대에 **'판매 장벽(Sell Wall)'**을 형성하여 시세를 통제하기도 한다.
* **바트 패턴(Bart Pattern):** 유동성이 낮은 시장에서 급격한 수직 상승 후 횡보하다가 다시 수직 하락하는 차트 모양이다. 이는 시장의 깊이가 얇을 때 소수의 대량 주문으로 가격이 쉽게 조작됨을 보여준다.



## 4. 커스터디(Custody) 인프라 설계 (개발자 핵심 지식)

거래소나 지갑 서비스 개발 시 자산의 안전한 보관은 최우선 과제다.

* **지갑 계층 구조:**
    * **핫 월렛(Hot Wallet):** 인터넷에 연결되어 즉시 출금이 가능한 지갑이다. 편의성은 높으나 해킹 위험이 상존한다.
    * **콜드 스토리지(Cold Storage):** 개인키를 오프라인 상태로 보관한다. 물리적 서명 절차가 필요하여 보안성이 극대화되나 처리 속도가 24~48시간으로 느리다.
    * **웜 월렛(Warm Wallet):** 핫과 콜드의 절충안으로, 화이트리스트 주소로만 자금을 분배하는 등 제한적 온라인 상태를 유지한다.
* **보안 규칙:**
    * **5% 미만의 규칙:** 전체 고객 자산의 95% 이상은 콜드 스토리지에, 5% 미만만 핫 월렛에 두어 피해를 최소화한다.
    * **화이트리스팅:** 예금 지갑이 탈취되더라도 자금이 미리 정해진 내부 주소로만 흐르도록 강제한다.





## 5. 차익 거래(Arbitrage)와 시스템 설계

거래소 간 가격 차이를 이용하는 차익 거래 시스템 구축 시 고려해야 할 기술적 제약이다.

* **삼각 차익 거래:** 동일 거래소 내에서 세 개 이상의 시장(예: USD→BTC→ETH→USD)을 거쳐 이익을 내는 방식이다.
* **플로트(Float) 관리:** 각 거래소에 자금을 분산 배치하여 전송 시간 지연 없이 즉시 체결하는 능력이 핵심이다.
* **뱅킹 및 전송 리스크:** 피아트(법정화폐) 송금은 은행 시스템을 타므로 며칠이 소요될 수 있다. 또한 암호화폐별 컨펌(Confirm) 속도(BTC 약 60분, XRP 약 1분 미만)에 따른 가격 변동 위험을 계산해야 한다.



## 6. 거래소 API 및 트레이딩 봇 개발 실무

* **REST API (Pull):** 트레이딩 봇이 주기적으로 데이터를 요청(Polling)한다. 요청 제한(Rate Limit) 문제와 데이터 지연(Latency)이 발생할 수 있다.
* **웹 소켓 (WebSocket, Push):** 시장에 변화가 생기면 서버가 즉시 데이터를 밀어준다. 실시간성이 중요한 고성능 봇 개발에 필수적이다.
* **보안 인증:** OAuth2 또는 HMAC 서명을 사용하여 API 키의 보안을 확보한다.
* **CCXT 라이브러리:** 수많은 거래소의 서로 다른 API 규격을 하나의 표준 인터페이스로 통합해주는 오픈소스 라이브러리다. 125개 이상의 거래소 연동을 지원한다.
* **샌드박스(Sandbox):** 실제 자금을 투입하기 전 가상 자산으로 로직을 테스트할 수 있는 환경이다. API 품질 평가 시 중요한 척도가 된다.



## 7. 마켓 데이터 분석 및 추적

* **블록 익스플로러:** 체인상의 모든 트랜잭션을 실시간으로 모니터링한다(Etherscan 등). 개발 시 지갑 간 자금 흐름을 추적하거나 입금 확인 자동화 로직을 짤 때 사용한다.
* **분석 방법:**
    * **기본적 분석:** 프로젝트의 로드맵, 사용자 수, 규제 현황 등 본질적 가치를 분석한다.
    * **기술적 분석:** 과거 가격과 거래량 데이터를 기반으로 차트 패턴을 분석한다.



### 결론 및 제언
블록체인 시스템 개발자에게 마켓 인프라는 **신뢰성(Reliability)**과 **보안(Security)**의 총체다. 단순히 API를 연결하는 수준을 넘어, **슬리피지 보정 로직, 가스비 및 거래 수수료 계산, 핫/콜드 월렛 간 자동 분배 시스템** 등을 정교하게 설계해야 한다. 특히 규제 관할권에 따른 트래블 룰(Travel Rule) 및 KYC/AML 대응 로직은 엔터프라이즈급 서비스 개발 시 필수적인 요소다.
