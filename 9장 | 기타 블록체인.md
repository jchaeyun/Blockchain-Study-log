# 9장| 기타 블록체인

비트코인과 이더리움이 연 탈중앙화의 시대는 이제 기업용 블록체인과 분산 원장 기술(DLT)로 확장되고 있다.



## 1. 엔터프라이즈 블록체인의 핵심 가치

기업은 투기적 자산보다는 블록체인의 기술적 특성 자체에 집중한다. DApp 개발 시 단순한 데이터베이스 대신 블록체인을 써야 하는 이유는 다음과 같다.

* **합의(Consensus):** 새로운 데이터의 유효성을 검사해 무결성을 유지한다.
* **해싱(Hashing):** 레코드를 통합해 수정 불가능한 감사 추적(Audit Trail)을 생성한다.
* **암호화(Encryption):** 데이터 전송과 접근 권한을 안전하게 보호한다.
* **분산화(Distribution):** 여러 당사자가 중앙 관리자 없이 동일한 데이터를 공유한다.


## 2. 데이터베이스 vs 분산 원장(DLT)

DApp 개발자는 기존의 관계형 데이터베이스(RDBMS)와 분산 원장의 차이를 명확히 인지해야 한다.

* **데이터베이스:** 중앙집중식이며 관리자가 액세스를 제어한다. 복제 시스템을 통해 가용성을 높이지만, 결국 관리자의 권한에 종속된다.
* **분산 원장(DLT):** 여러 당사자가 중앙 관리자 없이 정보에 액세스하고 업데이트한다. 
* **퍼미션리스(Permissionless):** 누구나 참여 가능한 공개형(비트코인, 이더리움). 인센티브 구조가 필수적이다.
* **퍼미션드(Permissioned):** 승인된 참가자만 참여하는 기업용. 인센티브(토큰)는 선택 사항이다.





## 3. 주요 엔터프라이즈 프레임워크

기업용 DApp을 개발할 때 선택할 수 있는 대표적인 플랫폼들이다.

### 하이퍼레저 패브릭 (Hyperledger Fabric)
* **특징:** 리눅스 재단 주도. Go, Java 등을 지원한다.
* **구조:** 모듈식 아키텍처로 합의 알고리즘을 선택할 수 있으며, 기업 환경에 맞춘 세밀한 권한 관리가 가능하다.

### 코다 (Corda)
* **특징:** 금융권 컨소시엄 R3 개발. JVM(Java, Kotlin) 기반이다.
* **DApp 개발 핵심:** 모든 참가자에게 데이터를 브로드캐스트하는 대신, 거래 당사자 간에만 데이터를 공유(Point-to-Point)한다. 이는 프라이버시가 중요한 금융 거래에 최적화된 설계다. 
* **합의 방식:** 타당성 합의(서명 및 계약 제약 확인)와 고유성 합의(이중 지출 방지)로 나뉜다.

### 쿼럼 (Quorum)
* **특징:** JP모건 개발. 이더리움 기반이므로 솔리디티(Solidity) 개발자에게 익숙하다.
* **프라이버시:** 비대칭키 암호화를 사용하여 특정 당사자만 트랜잭션 내용을 볼 수 있는 기능을 제공한다.



## 4. 프라이버시 강화 기술 (영지식 증명)

퍼블릭 체인의 투명성은 기업에 독이 될 수 있다. 이를 해결하기 위한 기술적 대안이다.

* **나이트폴(Nightfall):** EY 개발. 영지식 증명을 사용해 이더리움 상에서 프라이버시를 유지하며 토큰을 전송한다.
* **조크라테스(ZoKrates):** 이더리움에서 영지식 증명(ZK-SNARKs)을 쉽게 사용할 수 있도록 돕는 툴박스다.
* **영지식 증명(Zero-Knowledge Proofs):** 실제 데이터를 공개하지 않고도 그 데이터가 진실임을 증명하는 기술이다.



## 5. 산업별 실전 사용 사례

DApp의 비즈니스 모델을 구상할 때 참고할 만한 영역들이다.

* **뱅킹:** 국제 송금(JPM 코인), 디지털 채권 발행(산탄데르), 중앙은행 디지털 통화(CBDC).
* **공급망 및 물류:** 송장과 실제 화물 이동의 일치 여부를 스마트 컨트랙트로 자동 검증(비자 B2B 커넥트).
* **게임:** 아이템의 고유성 보장 및 위변조 방지.
* **헬스케어:** 분산된 환자 의료 기록의 통합 관리 및 환자의 데이터 주권 확보.
* **IoT:** 수많은 기기 간의 인증 및 소액 결제 자동화(IBM 왓슨 연동).



## 6. 리브라(Libra/Diem) 프로젝트와 Move 언어

페이스북(메타) 주도로 시작된 리브라는 기업용 블록체인의 대규모 배포 가능성을 보여준다.

### 기술적 특징
* **LBFT 합의:** 리더 노드를 선출하여 빠르게 투표를 진행하는 비잔틴 장애 허용 알고리즘이다. 초당 1,000건 이상의 트랜잭션을 처리한다.
* **무브(Move) 언어:** 리브라를 위해 개발된 새로운 프로토콜 언어다. 자산을 '리소스(Resource)'로 취급하여 복사나 소실을 언어 차원에서 방지하는 구조를 갖췄다. DApp 개발자에게는 솔리디티와는 다른 새로운 안전 지향적 패러다임을 제시한다.
* **노드 구조:** 트랜잭션을 처리하는 '유효성 검사기 노드(허가형)'와 상태를 브로드캐스트하는 '풀 노드(누구나 실행 가능)'로 나뉜다.

### 리브라 트랜잭션 구조 (개발 데이터 세트)
| 필드 | 설명 |
| :--- | :--- |
| **Sender address** | 보낸 사람의 계정 주소 |
| **Program** | Move 스크립트 바이트코드 및 입력 목록 |
| **Gas price** | microlibra 단위로 지불할 가스 가격 |
| **Sequence number** | 이중 지출 방지를 위한 계정별 일련번호 |





## 요약

9장은 블록체인이 투기를 넘어 기업의 운영 효율성을 높이는 '신뢰의 인프라'로 변모하는 과정을 보여준다. DApp 개발자는 다음과 같은 통찰을 얻어야 한다.

1.  데이터의 성격에 따라 퍼미션드(Corda, Fabric)와 **퍼미션리스(Ethereum)** 중 최적의 환경을 선택해야 한다.
2.  프라이버시가 중요하다면 **영지식 증명**이나 **쿼럼의 프라이빗 콘트랙트**를 활용해야 한다.
3.  BaaS(Azure, IBM, AWS 등)를 활용하면 노드 구축 부담을 줄이고 비즈니스 로직 개발에 집중할 수 있다.
4.  **Move**와 같은 자산 최적화 언어의 등장은 스마트 컨트랙트의 보안성과 안전성을 한 단계 높이는 지표가 된다.
